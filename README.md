# n8n на VPS через Docker Compose

Состав стека (наружу открыты только порты 80/443):

```
Интернет ──> Caddy (80/443, автоматический HTTPS) ──> n8n (5678, только внутри) ──> PostgreSQL
```

Вход в n8n — по логину (email) и паролю: при первом открытии страницы n8n
предложит создать аккаунт владельца. Дополнительно можно включить 2FA.

## Быстрый старт (одной командой)

С локальной машины скопируйте папку на сервер и запустите установщик:

```bash
scp -r . root@IP_СЕРВЕРА:/opt/n8n
ssh root@IP_СЕРВЕРА
cd /opt/n8n && sed -i 's/\r$//' setup.sh backup.sh && sudo bash setup.sh
```

`setup.sh` сам установит Docker, создаст `.env` со сгенерированными секретами
(спросит только домен или IP), настроит файрвол, запустит контейнеры и
доустановит community-ноды из списка `NODES` в начале скрипта.
Разделы 2–6 ниже нужны только для ручной установки без скрипта.

## 1. Требования

- VPS с Ubuntu 22.04/24.04 (или другой Linux), от 2 ГБ RAM
- SSH-доступ root или sudo
- Для варианта с HTTPS — любой домен/поддомен, A-запись которого указывает на IP сервера

## 2. Установка Docker на сервере

```bash
curl -fsSL https://get.docker.com | sh
docker compose version   # проверка
```

## 3. Загрузка файлов проекта на сервер

С локальной машины (PowerShell/cmd, из этой папки):

```bash
scp -r . root@IP_СЕРВЕРА:/opt/n8n
```

Дальше все команды выполняются на сервере в каталоге `/opt/n8n`.

## 4. Конфигурация

```bash
cd /opt/n8n
cp .env.example .env
chmod 600 .env

# Сгенерировать секреты и вписать их в .env:
openssl rand -hex 32   # -> N8N_ENCRYPTION_KEY
openssl rand -hex 16   # -> POSTGRES_PASSWORD
nano .env
```

В `.env` выберите вариант доступа:

- **Вариант A (рекомендуется)** — есть домен: укажите его в
  `N8N_SITE_ADDRESS`, `N8N_HOST`, `WEBHOOK_URL`. Caddy сам получит сертификат
  Let's Encrypt и будет продлевать его. Многие сервисы (например, Telegram)
  принимают вебхуки только по HTTPS, поэтому для интеграций домен почти
  обязателен.
- **Вариант B** — только IP: закомментируйте блок A, раскомментируйте блок B,
  подставьте IP сервера. Трафик (включая пароль при входе) не шифруется —
  используйте только как временное решение.

⚠️ Сохраните `N8N_ENCRYPTION_KEY` в надёжном месте (менеджер паролей).
Без него резервная копия не позволит восстановить сохранённые credentials.

Права на папку обмена файлами (контейнер n8n работает от uid 1000):

```bash
chown -R 1000:1000 local-files
```

Если файлы редактировались в Windows, поправьте переносы строк у скрипта:

```bash
sed -i 's/\r$//' setup.sh backup.sh && chmod +x setup.sh backup.sh
```

## 5. Файрвол

```bash
ufw allow OpenSSH
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable
```

## 6. Запуск

```bash
docker compose up -d
docker compose logs -f n8n    # ждём строку "Editor is now accessible"
```

Откройте `https://ваш-домен` (вариант A) или `http://IP-сервера` (вариант B).

## 7. Первый вход

1. n8n покажет форму **Set up owner account** — задайте email, имя и пароль.
   Это и есть ваш логин/пароль; форма доступна только один раз, сразу после
   установки, поэтому создайте аккаунт сразу.
2. Рекомендуется включить 2FA: **Settings → Personal → Two-factor authentication**.
3. Если нужен второй рубеж защиты перед самим n8n — в `Caddyfile` есть
   закомментированный блок `basic_auth` с инструкцией.

## 8. Обновление n8n

```bash
cd /opt/n8n
./backup.sh                # сначала бэкап
docker compose pull
docker compose up -d
```

По умолчанию используется последняя стабильная версия. Чтобы зафиксировать
конкретную, укажите тег в `docker-compose.yml`, например
`image: docker.n8n.io/n8nio/n8n:1.97.0`.

## 9. Резервные копии

Разовый запуск: `./backup.sh` — создаст в `./backups` дамп БД и архив данных
n8n, копии старше 14 дней удаляются автоматически.

Автоматически каждую ночь в 03:00:

```bash
crontab -e
# добавить строку:
0 3 * * * cd /opt/n8n && ./backup.sh >> /opt/n8n/backups/backup.log 2>&1
```

Восстановление БД из дампа (на работающем стеке, workflow остановлены):

```bash
gunzip < backups/n8n-db-ДАТА.sql.gz | docker compose exec -T postgres psql -U n8n -d n8n
docker compose restart n8n
```

Архив `n8n-data-*.tar.gz` содержит `/home/node/.n8n` (конфиг с ключом
шифрования и бинарные данные) — при переезде на новый сервер распакуйте его в
том `n8n_data` и используйте тот же `N8N_ENCRYPTION_KEY` в `.env`.

## 10. Community-ноды (кастомные ноды)

Ставятся прямо из интерфейса: **Settings → Community nodes → Install** →
ввести имя npm-пакета (например, `n8n-nodes-mcp`) → подтвердить → Install.
Перезапуск контейнера не требуется. Использование community-нод как
инструментов AI-агентов уже разрешено в `docker-compose.yml`
(`N8N_COMMUNITY_PACKAGES_ALLOW_TOOL_USAGE=true`).

При установке через `setup.sh` стартовый набор нод ставится автоматически —
список задаётся массивом `NODES` в начале скрипта. Чтобы добавить пакет позже,
допишите его в список и выполните на сервере:

```bash
bash setup.sh nodes
```

Ноды ставятся в `/home/node/.n8n/nodes` (том `n8n_data`), поэтому переживают
обновление образа n8n. После установки они появляются в палитре нод редактора.

Community-ноды выполняют сторонний код на вашем сервере — ставьте только
проверенные пакеты (рейтинг: https://github.com/restyler/awesome-n8n).

## 11. Частые проблемы

| Симптом | Решение |
|---|---|
| При входе ошибка про «secure cookie» | Вы на варианте B: проверьте `N8N_SECURE_COOKIE=false` в `.env`, затем `docker compose up -d` |
| Сертификат не выпускается | Проверьте, что A-запись домена указывает на сервер и порты 80/443 открыты: `docker compose logs caddy` |
| Забыт пароль владельца | `docker compose exec n8n n8n user-management:reset` — вернёт мастер первичной настройки, данные сохранятся |
| Вебхуки не срабатывают извне | `WEBHOOK_URL` в `.env` должен совпадать с реальным внешним адресом |
| Нода не может писать в `/files` | На сервере: `chown -R 1000:1000 local-files` |

## 12. Свои TLS-сертификаты (вместо Let's Encrypt)

По умолчанию при указанном домене Caddy получает сертификат Let's Encrypt
автоматически — ничего делать не нужно. Если есть собственные сертификаты,
всё подключается одной командой на сервере:

```bash
bash setup.sh certs n8n.example.com /путь/к/fullchain.pem /путь/к/privkey.pem
```

Скрипт скопирует файлы в `certs/`, проверит их (PEM-формат, соответствие
ключа сертификату, домен, срок действия, наличие цепочки CA), включит `tls`
в `Caddyfile` и монтирование папки в `docker-compose.yml`, переключит `.env`
на HTTPS-режим и перезапустит Caddy.

- Если файлы уже лежат в `certs/` под именами `fullchain.pem` и `privkey.pem`,
  пути в команде можно не указывать.
- При полной установке (`sudo bash setup.sh домен`) скрипт сам найдёт файлы
  в `certs/` и включит их вместо Let's Encrypt.
- Продление — та же команда с новыми файлами.

Форматы: `fullchain.pem` — сертификат домена + промежуточная цепочка CA.
Если выданы отдельные `domain.crt` и `ca-bundle.crt`, склейте:
`cat domain.crt ca-bundle.crt > fullchain.pem`. Если выдан PFX:
`openssl pkcs12 -in cert.pfx -clcerts -nokeys -out fullchain.pem` и
`openssl pkcs12 -in cert.pfx -nocerts -nodes -out privkey.pem`.

Самоподписанные сертификаты для Telegram-триггера не подойдут — Telegram
требует доверенный публичный CA.
